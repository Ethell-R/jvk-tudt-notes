  1. A typical web application

    A typical web application

    A typical web application involves a server which

    • serves content to the client
    • runs the web application (business/domain) logic
    • handles the application state with the use of some type of data storage, most commonly a database

  2. Going serverless with BaaS

    Going serverless with BaaS

    However, it is possible to develop a full web application using only HTML, CSS and Javascript, by using BaaS:

    • it stands for Backend as a Service (also called Mobile Baas - MBaaS - when referred to in the context of mobile native development)
    • is accessed using a Web-exposed API, offering services such as
      • authentication
      • database
      • cloud storage
      • push notifications
      • API building (codeless function definition)
      • user-defined functions (also called Function as a Service - FaaS)
    • is one of the many different technologies that are referred to as 'serverless' - these include F(unction)aaS and C(ontainer)aaS
    • providers are e.g. Firebase (Google) and AWS Amplify
    • Parse is an implementation that needs to be self-hosted - owned by Parse (2011), then Facebook (2013), then made open-source (2017)

  3. What does this have to do with RWAT?

    What does this have to do with RWAT?

    • BaaS allows us to develop full-stack applications with the use of client-side technologies only.
    • While BaaS does not provide the flexibility of design and range of control that an own server would provide
    • it also does not incurr the cost and effort associated with running a full-blown server.
